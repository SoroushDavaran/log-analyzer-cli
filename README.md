
# 📊 Access Log Analyzer (CLI Tool)

یک ابزار قدرتمند، سریع و سبک در خط فرمان (CLI) برای پارس و تحلیل فایل‌های `access log` با فرمت استاندارد **Combined Log Format**. 

این پروژه با هدف شبیه‌سازی دقیق اولین قدم یک مهندس زیرساخت/DevOps هنگام بروز اختلال یا کندی در سرویس‌ها طراحی شده است؛ رفتن به سراغ لاگ‌های خام، کثیف و حجیم و استخراج سریع گزارش‌های حیاتی بدون اشغال حافظه سرور.

---

## ✨ ویژگی‌های کلیدی (Features)

### 🔹 تحویل‌دادنی‌های الزامی (Core)
*   **پارس ایمن و مقاوم (Defensive Parsing):** استفاده از Regex بهینه‌سازی‌شده برای مدیریت خطوط خراب یا ناقص بدون کرش کردن ابزار.
*   **پردازش خط‌به‌خط (Streaming Iterator):** خواندن فایل با الگوی در جریان (`for line in f`)؛ تضمین اجرای روان روی فایل‌های چند گیگابایتی با پیچیدگی حافظه ثابت $O(1)$.
*   **گزارش آماری پایه:** نمایش تعداد کل درخواست‌ها، IPهای منحصربه‌فرد، نرخ خطای کل ($4xx + 5xx$) و حجم کل بایت‌های ارسالی.
*   **توزیع زمانی ترافیک:** نمایش توزیع ترافیک به تفکیک ساعت به‌همراه یک **هیستوگرام متنی** شکیل برای شناسایی سریع پیک‌های ترافیکی.

### 🔸 تحویل‌دادنی‌های امتیازی (Bonus)
*   **پشتیبانی بومی از Gzip:** قابلیت خواندن مستقیم فایل‌های فشرده با پسوند `.gz` بدون نیاز به اکسترکت دستی.
*   **تشخیص هوشمند Brute-Force:** شناسایی IPهای مشکوکی که تعداد پاسخ‌های `401 Unauthorized` آن‌ها روی مسیرهای حساس (مانند `/login`) از آستانه مجاز فراتر رفته است.
*   **الگوریتم خودکار تشخیص Spike خطا:** شناسایی خودکار ساعت‌هایی که نرخ خطای `5xx` در آن‌ها جهش ناگهانی نسبت به میانگین کل (Baseline) داشته است.
*   **خروجی دوگانه (Format Agnostic):** پشتیبانی از خروجی متنی خوانا برای انسان (Human-Readable) یا خروجی ساختاریافته `JSON` برای پایپ کردن به ابزارهای دیگر (مثل jq).
*   **فیلترهای پیشرفته زمانی:** قابلیت فیلتر کردن دقیق لاگ‌ها بر اساس بازه زمانی مشخص (`--start` و `--end`).
*   **تست‌های واحد (Unit Tests):** پوشش تست برای توابع پارسر و توابع تجمیع آمار.

## 🚀 راهنمای نصب و اجرا

### پیش‌نیازها
این ابزار کاملاً متکی بر **کتابخانه استاندارد پایتون (Standard Library)** است و به هیچ پکیج جانبی (`pip install`) نیاز ندارد. تنها پیش‌نیاز، نصب بودن **Python 3.10** یا بالاتر (به دلیل بهره‌گیری از Type Hintهای جدید) است.

### دستورات نمونه جهت اجرا:

```bash
# ۱. اجرای ساده روی لاگ متنی
python log_analyzer.py sample_access.log

# ۲. اجرا روی فایل فشرده (بدون Unzip دستی)
python log_analyzer.py sample_access.log.gz

# ۳. دریافت خروجی با فرمت ماشین (JSON)
python log_analyzer.py sample_access.log --json

# ۴. شخصی‌سازی تعداد مسیرهای پر ترافیک برتر (مثلاً ۵ مسیر برتر)
python log_analyzer.py sample_access.log --top 5

# ۵. فیلتر کردن لاگ‌ها در یک بازه زمانی خاص
python log_analyzer.py sample_access.log --start 2026-06-01T14:00:00 --end 2026-06-01T15:30:00

# ۶. تنظیم دقیق پارامترهای امنیتی و تشخیص خطا
python log_analyzer.py sample_access.log --suspicious-threshold 5 --spike-multiplier 2.5 --login-path signin

```

### 🛠️ ابزارهای توسعه‌دهنده (تست و دمو)

```bash
# تولید فایل لاگ نمونه با تعداد خطوط دلخواه
python generate_sample_log.py --lines 300000 --out sample_access.log

# اجرای تست‌های واحد با جزییات کامل (Verbose)
python -m unittest test_log_analyzer.py -v

```

برای مشاهده تمامی گزینه‌ها و راهنما، دستور زیر را اجرا کنید:

```bash
python log_analyzer.py --help

```

## 🧠 تصمیمات فنی و معماری (Engineering Design)

* **Regex Pre-compilation:** کامپایل یک‌بارۀ الگوی لاگ (`LOG_PATTERN`) در سطح ماژول انجام شده است. استفاده از Regex به جای متد `split` به این دلیل است که فیلدهای داخل کوتیشن (مثل User-Agent یا Referrer) ممکن است خود حاوی فضا (Space) باشند که متد اسپلیت ساده را کاملاً شکننده می‌کند.


* **Single-Pass Algorithm:** تمامی آمارها، کلاس‌های وضعیت، هیستوگرام‌های ساعتی و الگوهای اتک تنها در **یک‌بار عبور** از روی فایل محاسبه می‌شوند و فایل هرگز مجدداً بازخوانی یا اصطلاحاً Seek نمی‌شود.


* **تغییر وضعیت خودکار اندازه:** در فرمت کامبایند، کاراکتر `-` به معنی عدم وجود بدنه (Body) است که سیستم به صورت خودکار آن را به مقدار `0` بایت تبدیل می‌کند تا محاسبات آماری مختل نشود.


* **محدودیت شناخته شده و فضا برای بهبود:** در بخش `top_endpoints` مسیرها دقیقاً به همان صورت خام (همراه با Query Stringها) شمرده می‌شوند. برای تجمیع مسیرهای یکسان (مثلاً یکی کردن `/p?id=1` و `/p?id=2`) می‌توان قبل از افزایش کانتر، بخش کوئری استرینگ را با متد اسمارت اسپلیت حذف کرد.

## 🛠️ چالش مهندسی و راه‌حل (Problem & Solution)

**چالش:** در فاز طراحی بخش "تشخیص خودکار بازه جهش خطا"، نسخه ابتدایی سیستم صرفاً ساعت‌هایی را که نرخ خطای ۵۰۰ آن‌ها از یک عدد ثابت (مثلاً ۱۰٪) فراتر می‌رفت گزارش می‌کرد. اما در سناریوهای واقعی این منطق شکننده است؛ زیرا یک سرویس با خطای پایه‌ی همیشگیِ ۵٪ با کوچک‌ترین نوسان آلارم کاذب می‌داد، در حالی که یک سرویس پایدار با خطای پایه‌ی ۰.۰۱٪ اگر ناگهان به ۴٪ خطا می‌رسید (که یک فاجعه است) شناسایی نمی‌شد.

**راه‌حل:** منطق سیستم به **نرخ نوسان پویا نسبت به میانگین کل (Dynamic Baseline)** تغییر یافت. در پیاده‌سازی نهایی، ابزار ابتدا میانگین کل خطای ۵xx فایل را محاسبه کرده و سپس ساعت‌هایی را که نرخ خطایشان حداقل $X$ برابر (بر اساس آرگومان `--spike-multiplier`) میانگین کل باشد گزارش می‌دهد. همچنین پارامتر `min_requests` لحاظ شد تا ساعت‌های کم‌ترافیک (مثلاً نیمه‌شب با تک درخواست‌های تصادفی) به اشتباه به عنوان Spike شناسایی نشوند.

```

### چند نکته تکمیلی برای مصاحبه:
* از آنجا که در کد از الگوی بسیار عالی شی‌گرایی (`LogAnalyzer`) استفاده کرده‌اید و متدها مجزا هستند، در مصاحبه حتماً روی این موضوع تأکید کنید که کد شما به راحتی قابل گسترش (Extensible) است؛ مثلاً اضافه کردن فرمت‌های لاگ دیگر (مثل Nginx JSON log یا کلاسترینگ IPها) بدون دست زدن به بدنه اصلی CLI ممکن است.
```
